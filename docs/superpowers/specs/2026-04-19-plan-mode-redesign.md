# Plan Mode Redesign â€” Role-Based Tool Access + Pre-Execution Planning Gates

**Date:** 2026-04-19  
**Status:** Design Review  
**Scope:** rustycode-orchestra (plan mode) + rustycode-core (team coordination)

---

## Executive Summary

Current plan mode is disconnected from execution. It's a state machine that checks tool access but isn't integrated with actual tool gating, multi-agent coordination, or the existing Coordinator/Builder/Skeptic/Judge team model.

**Redesign:** Replace the global `ExecutionPhase` (Planning/Implementation) with **role-based tool access** paired with **pre-execution planning gates tied to convoys**. Agents operate based on their role; convoys gate whether the Coordinator (execution loop) can start.

**Key changes:**
1. Remove global execution phases; add role-based tool matrices
2. Move planning to a separate orchestration phase (Planner agents write plans)
3. Plan becomes a first-class input to Coordinator execution
4. Convoys track approval gates and plan state
5. Tool access determined by: `agent_role` + `convoy_plan_approved`

---

## Problem Statement

### Current Issues

1. **Disconnected state machine**: `PlanMode` in orchestra tracks phases but doesn't gate actual tool execution
2. **Tool access unchecked**: `is_tool_allowed()` exists but isn't called in tool executors
3. **No multi-agent coordination**: Agents can't work in parallel phases (planner on task N+1, worker on task N)
4. **Mock plan generation**: `AutoMode::generate_plan()` is hardcoded, not LLM-driven
5. **Dual state tracking**: TUI layer and orchestra layer both track plan mode separately
6. **Missing tools**: Some planning tools (bash, write, agents) not in allowed lists
7. **Disconnected from team model**: Orchestra's simple roles don't map to core's Coordinator/Builder/Skeptic/Judge

### What We Want

- **Explicit planning phase**: Agents analyze and write plans before execution
- **Approval gates**: Complex plans require approval; simple plans auto-approve
- **Parallel execution**: Multiple agents can work independently (planner on next task, executor on current)
- **Real LLM planning**: Plans generated by actual model calls with structured output
- **Role-based access**: What you can do depends on your role, not a global phase
- **Integrated with core team model**: Planning feeds naturally into Coordinator loop
- **Proper tool gating**: Forbidden tools actually blocked at executor level

## Architecture

### 1. Agent Roles & Tool Access Matrix

Replace the global `ExecutionPhase` with **role-based tool matrices**. Each agent has a role; role determines allowed tools.

#### AgentRole (from `rustycode-core::team` / `rustycode-orchestra::agent_identity`)

Extend existing `AgentRole` enum to clarify tool access:

```rust
pub enum AgentRole {
    /// Analyzes code, writes plans, researches solutions
    Planner,
    /// Executes approved plans, makes code changes
    Worker,
    /// Reviews changes, tests, verifies correctness
    Reviewer,
    /// Explores codebase, gathers context
    Researcher,
    /// Leads team, coordinates phases (future)
    Coordinator,
}
```

#### Tool Access Matrix

**Planner agents** can:
- Read: `read`, `read_file`, `grep`, `glob`, `list_dir`, `lsp`, `web_search`, `web_fetch`
- Write: `write`, `edit_file` (for temporary/test files)
- Execute: `bash` (for exploration, not production changes)
- Dispatch: `Agent` (research, exploration)
- Plan: `TaskCreate`, `TaskList`, `TaskUpdate` (structure the plan)

**Worker agents** can:
- Read: `read`, `read_file`, `grep`, `glob`, `list_dir`, `lsp`
- Write: `write`, `edit_file`, `apply_patch`
- Execute: `bash`
- Blocked: `Agent` dispatch, `TaskCreate` (follows plan, doesn't create new tasks)
- Gate: Only after convoy plan is approved

**Reviewer agents** can:
- Read: `read`, `read_file`, `grep`, `glob`, `lsp`, `web_fetch`
- Execute: `bash` (test-only)
- Blocked: Any file writes, `Agent` dispatch

**Researcher agents** can:
- Read: `read`, `read_file`, `grep`, `glob`, `lsp`, `web_search`, `web_fetch`
- Dispatch: `Agent` (other researchers, explorers)
- Blocked: Any writes, execute commands

### 4. Data Flow Integration

#### PlanMode (refactored in `rustycode-orchestra/src/plan_mode.rs`)

Remove `ExecutionPhase`, `current_phase`, `set_phase()`. Keep:

```rust
pub struct PlanMode {
    /// Map of role â†’ allowed tools
    role_tool_matrix: HashMap<AgentRole, HashSet<&'static str>>,
    
    /// Approval criteria
    approval_triggers: Vec<ApprovalTrigger>,
    
    /// Config (enabled, require_approval, etc.)
    config: PlanModeConfig,
}

impl PlanMode {
    /// Check if an agent with a given role can use a tool
    pub fn can_use_tool(&self, role: AgentRole, tool: &str) -> Result<(), ToolBlockedReason> {
        let allowed = self.role_tool_matrix
            .get(&role)
            .ok_or(ToolBlockedReason::UnknownRole(role))?;
        
        if allowed.contains(tool) {
            Ok(())
        } else {
            Err(ToolBlockedReason::NotAllowedForRole { tool, role })
        }
    }
    
    /// Compute whether a plan requires approval
    pub fn assess_approval_required(&self, plan: &ConvoyPlan) -> bool {
        for trigger in &self.approval_triggers {
            match trigger {
                ApprovalTrigger::HighRisk(level) => {
                    if plan.risks.iter().any(|r| r.level >= *level) {
                        return true;
                    }
                }
                ApprovalTrigger::HighCost { threshold, .. } => {
                    if plan.estimated_cost_usd > *threshold {
                        return true;
                    }
                }
                ApprovalTrigger::ExternalChanges { .. } => {
                    if plan.files_to_modify.iter().any(|f| is_external_path(&f.path)) {
                        return true;
                    }
                }
            }
        }
        false
    }
}
```

### 5. Tool Execution Gating

**Enforcement point**: Tool executors (bash, write, edit, etc.) check role-based access before execution.

In `rustycode-tools/src/bash.rs` (or tool executor):

```rust
pub async fn execute_bash(
    cmd: &str,
    agent_role: AgentRole,
    plan_mode: &PlanMode,
) -> Result<Output> {
    // Check role-based access
    plan_mode.can_use_tool(agent_role, "bash")?;
    
    // Validate command against sandbox rules
    validate_command(cmd)?;
    
    // Execute
    tokio::process::Command::new("sh")
        .arg("-c")
        .arg(cmd)
        .output()
        .await
        .context("bash execution failed")
}
```

### 6. Multi-Agent Parallelism

With this design, agents can work in parallel:

**Scenario: Task convoy with subtasks**
```
Convoy: feature-auth

Worker-1 (role=Worker):
  â””â”€ Executing plan for task-1 (auth-module refactor)

Planner-1 (role=Planner):
  â””â”€ Writing plan for task-2 (auth-config) in same convoy

Reviewer-1 (role=Reviewer):
  â””â”€ Reviewing Worker-1's changes from task-1
```

No phase conflicts because:
- Planner's tool access doesn't depend on global phase
- Worker's access gated by convoy plan approval (not global phase)
- Reviewer always has same access regardless

## Approval Criteria

Plan requires explicit user approval if **any** of:

1. **High-risk changes**
   - FileAction::Delete in plan
   - Refactoring with "major" risk level or higher
   
2. **High estimated cost**
   - `estimated_cost_usd > config.cost_threshold` (default: $1.00)
   
3. **External system changes**
   - Modifying: config files, API clients, infrastructure, database schemas
   - Examples: `.env`, `secrets.json`, Dockerfile, docker-compose.yml

**Auto-approve** if:
- Low/medium risk
- Low cost (<$0.10)
- Internal code only

## Risks & Mitigations

| Risk | Mitigation |
|------|-----------|
| Breaking change to AutoMode | Deprecate old API, provide compatibility layer |
| Coordinator overload with plan context | Plan is optional; Coordinator works without it |
| Approval bottleneck | Auto-approve low-risk plans; make approval process fast |
| Agents blocked by missing tools | Audit and add missing tools to role matrices |
| Plan becomes stale during execution | Builder can escalate if plan becomes invalid |
